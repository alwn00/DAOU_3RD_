# DAOU AX 플레이그라운드 — 동작별 백엔드 동작 설명서

> 발표용. "내가 화면에서 어떤 동작을 하면, 뒤에서 백엔드가 어떻게 도는가"를 흐름별로 정리했습니다.
> 백엔드는 `backend/`의 FastAPI 상태머신입니다. 프론트는 **라이브 모드 토글**로 백엔드와 연결됩니다.

## 읽는 법 (범례)

| 표시 | 의미 |
|---|---|
| 🟢 **[실제 백엔드]** | 라이브 모드 ON일 때 실제 API를 호출하고 **SQLite DB에 기록**됩니다. (시연 시 진짜로 서버에서 일어남) |
| ⚪ **[프론트 목업]** | 지금은 화면 안에서만 동작(백엔드 미연동). 옆에 **확장 시 연결 지점**을 함께 표기 |

**두 개의 스위치를 먼저 이해하세요.**
- **로그인(모의 SSO)** — 내가 *누구인지*(이메일·부서·역할)를 정합니다. 이 정체성이 이후 모든 백엔드 호출의 `author`/`actor_role`/`user` 값으로 흘러갑니다.
- **라이브 모드 토글(좌하단 🔌)** — OFF면 전부 목업 연출, ON이면 등록·심사·가져가기가 실제 백엔드로 처리됩니다.

**핵심 한 줄:** 이 백엔드의 존재 이유는 *버튼 결과*가 아니라 **자산의 상태 전이가 실제 DB(`StatusHistory`)에 남는 상태머신**을 보여주는 것입니다.

### 상태머신 (모든 흐름의 뼈대)
```
DRAFT → AUTO_VALIDATING → (통과) → TRACK_ASSIGNED
  ├─ LIGHT → AUTO_APPROVED → PUBLISHED                        (프롬프트·설정형 + L1/L2)
  └─ HEAVY → REVIEW_PENDING → (AX기획 승인)
                               ├─ 일반: → PUBLISHED
                               └─ 웹앱: → TECH_REVIEW → BUILD_SIM → DEPLOY_SIM → PUBLISHED
어느 단계든 → REJECTED(사유 필수)  ·  PUBLISHED → (신고 3회) → REVIEW_PENDING 재진입
```

### 한눈에 보기 (동작 → 백엔드)
| 사용자 동작 | 백엔드 | 엔드포인트 |
|---|---|---|
| 로그인 | ⚪ 없음(프론트 모의 SSO) | — (정체성만 주입) |
| 홈·자산 찾기 | ⚪ 목업 목록 | 확장: `GET /assets` |
| **AI 등록** | 🟢 상태머신 진입 | `POST /assets` |
| 같이 개발(공동 제작) | ⚪ 목업 | 확장: 등록 시 공동관리자 필드 |
| **사용하기(가져가기)** | 🟢 사용 로그+점수 | `POST /assets/{id}/take` |
| 개선·제안(신고) | 🟢 신고→재심사 | `POST /assets/{id}/report` |
| 교육 학습 | ⚪ 목업 | — |
| 우수작 따라하기 | ⚪ 목업 | 확장: `UsageLog(RUN/TRANSCRIBE)` |
| 나의 대시보드 | ⚪ 목업 집계 | 확장: `GET /scores`·`GET /stats` |
| **관리(기획팀) 심사** | 🟢 승인/반려 | `POST /assets/{id}/review` |
| **관리(개발팀) 빌드·배포** | 🟢 빌드 시뮬 | `POST …/review`(웹앱) + `GET /assets/{id}` 폴링 |

## 1. 로그인

**동작:** 로그인 화면에서 "Google Workspace 계정으로 계속" → daou.co.kr 계정 선택.

- **프론트가 하는 일:** 이메일 도메인 검증(`@daou.co.kr`만 허용 — hosted-domain 제한 모사) → 선택 계정의 `이메일·이름·부서·역할`을 세션(localStorage)에 저장하고 `ME`·액터·프로필에 주입.
- **백엔드:** ⚪ **없음.** PoC는 오프라인 모의 SSO라 인증 서버 왕복이 없습니다. 대신 **로그인한 정체성이 이후 호출의 값이 됩니다.**
  - 등록 시 `author`=로그인 이름, `dept`=로그인 부서
  - 심사 시 `actor_role`=로그인 역할(AX_PLAN/AX_DEV …)
  - 가져가기 시 `user`=로그인 이름
- **확장 시(실제 배포):** Google OIDC 리다이렉트 → 백엔드 `/auth/callback`에서 `id_token` 검증(`iss/aud/exp/hd=daou.co.kr`) → 세션 쿠키 발급. 이후 `actor_role`은 요청 body가 아니라 **세션에서 서버가 주입**. (상세: `backend/README.md` 8장)

> 즉, 로그인은 백엔드를 부르지 않지만 **"누가 했는가"를 백엔드 기록(작성자·심사자·사용자)에 새기는 출발점**입니다.

## 2. 시뮬레이션 (온보딩 여정)

> 처음 이용자에게 "예"를 받으면 코치마크로 여정을 안내합니다("아니오"면 바로 홈). 여정 각 단계에서 백엔드가 어디까지 관여하는지 정리합니다.

### 2-0. 웰컴 · "처음 이용해보시나요?"
⚪ **[프론트 목업]** 코치마크(안내 말풍선)만. 백엔드 호출 없음.

### 2-1. 홈 화면 소개
⚪ **[프론트 목업]** 홈의 자산 목록은 프론트 `ASSETS` 배열을 렌더. **확장 시 연결:** `GET /assets` (목록) → 서버의 PUBLISHED 자산으로 대체.

### 2-2. 자산 찾기 소개 (검색·카테고리)
⚪ **[프론트 목업]** 클라이언트 필터. **확장 시 연결:** `GET /assets?status=PUBLISHED&type=&dept=` (백엔드가 이미 이 쿼리 필터를 지원).

### 2-3. AI 등록 🟢 **[실제 백엔드] — 이 여정의 핵심**
**동작:** 등록 3단계 작성 → "등록 신청".

- **호출:** `POST /assets`
  ```json
  { "title":"인사 규정 안내 Gem", "type":"AGENT", "platform":"GEMINI",
    "data_level":"L2", "author":"박서연", "dept":"인사팀",
    "purpose":"사내 인사 규정 질문에 규정 문서 기반으로 답하는 설정형 에이전트입니다" }
  ```
  (프론트 매핑: 설정형→`AGENT`, 프롬프트→`PROMPT`, 자동화→`AUTOMATION`, 웹앱→`WEBAPP` / 민감정보→`L3`, 부서한정→`L2`, 전사공개→`L1`)
- **백엔드 내부 처리(자동, 한 번의 호출로 연쇄 진행):**
  1. `Asset` INSERT (status=**DRAFT**) + `StatusHistory` 기록
  2. **AUTO_VALIDATING** — 실제 정규식 검증: 필수 필드, `purpose` 최소 10자, 금칙 패턴(주민번호·전화·이메일·`password/api_key/secret`)
     - 걸리면 → **REJECTED**(사유 기록) + `AuditLog`. 여기서 종료.
  3. **TRACK_ASSIGNED** — 트랙 배정 룰: 프롬프트·설정형+L1/L2 → **LIGHT** / 자동화·웹앱 또는 L3 → **HEAVY**
  4. 유효 등록에 **Score +3**(공급 기여) 적립
  5. LIGHT → **AUTO_APPROVED → PUBLISHED**(즉시 공개) / HEAVY → **REVIEW_PENDING**(심사 대기)
- **DB에 남는 것:** `Asset`, 전 단계 `StatusHistory`, `AuditLog`, `Score`.
- **화면 증거:** 등록 진행 로그에 **실제 status_history가 순서대로 스트리밍**되고, "📜 백엔드 상태 이력 보기" 버튼으로 DB 기록 원문을 그대로 확인 → *"목업이 아니라 진짜 서버 전이"*.

### 2-4. 같이 개발 (공동 제작 / 제작 요청)
⚪ **[프론트 목업]** 제작 요청·공동 제작 참여는 화면 연출. **확장 시 연결:** 등록(`POST /assets`) 시 소유자·공동관리자·소유권 이전 필드를 추가(정책상 이미 확정된 구조). 채택 시 공급 점수 배분.

### 2-5. 사용하기 (가져가기) 🟢 **[실제 백엔드]**
**동작:** 상세 화면에서 "가져가기".

- **호출:** `POST /assets/{id}/take` — body `{ "user":"박서연" }`
- **백엔드 처리:** `UsageLog` INSERT(action=**TAKE**) → 점수 규칙 적용
  - 본인(작성자=사용자)이면 점수 없음 / 타인 최초 가져가기면 **작성자에게 +1**(동일인 1회, 자산당 보너스 상한 +3)
- **응답:** `{ "asset_id":1, "user":"박서연", "score_awarded_to_author": true|false }`
- **화면 증거:** "🔌 [LIVE] 백엔드 기록 — 제작자 +1점 / 점수 없음(본인·중복·상한)" 토스트. (라이브 등록본만 실제 호출, 기존 목업 카탈로그 자산은 목업 처리)

### 2-6. 개선 및 제안 (신고 / 수정 제안)
🟢 **[실제 백엔드] — 신고:** `POST /assets/{id}/report` — body `{ "reporter":"박서연", "reason":"부정확" }`
  - 백엔드: 신고 카운트 누적 → **3회 도달 시 PUBLISHED → REVIEW_PENDING 재진입**(사후 관리 루프) + `AuditLog`.
  - *현재 프론트는 신고 버튼이 라이브에 연결돼 있지 않습니다(엔드포인트·동작은 완성·테스트됨). 원하면 가져가기처럼 1개 함수로 연결 가능 — 확장 지점.*

⚪ **수정 제안·댓글:** 프론트 목업. **확장 시 연결:** 코멘트 테이블 + 자산 버전 증가.

### 2-7. 교육 듣고 학습
⚪ **[프론트 목업]** AI 교육 모음/실습은 화면 콘텐츠. 백엔드 없음. (교육 자산도 결국 `Asset`의 한 종류로 등록·공개 가능 — 확장 시 동일 상태머신 재사용)

### 2-8. 우수작 따라하기 (따라 만들기)
⚪ **[프론트 목업]** 원본을 참고해 내 초안을 만드는 학습 사다리. 원본 레코드는 불변. **확장 시 연결:** `UsageLog`(action=**RUN**/TRANSCRIBE) 기록으로 "따라 만든 횟수" 집계, 저장 시 새 `Asset` DRAFT 생성.

### 2-9. 나의 대시보드 확인
⚪ **[프론트 목업]** 점수·기여·활동을 프론트 `ledger`로 집계. **확장 시 연결:**
  - `GET /scores` → 공급 기여 점수판(사용자별 합계)
  - `GET /stats` → 등록 수·PUBLISHED 수·가져가기 수·트랙별/유형별 건수

## 3. 관리 — 기획팀 (AX_PLAN) 🟢 **[실제 백엔드]**

> 로그인 계정이 AX기획팀(최운영)일 때 AX 운영 콘솔이 열립니다.

**동작 A) 심사 큐 확인**
- 현재: 프론트 `REVIEW_QUEUE`(라이브 등록분은 백엔드 `beId` 보유). **확장 시 연결:** `GET /assets?status=REVIEW_PENDING`.

**동작 B) 승인 / 반려**
- **호출:** `POST /assets/{id}/review`
  ```json
  { "actor_role":"AX_PLAN", "decision":"APPROVE", "reason":"정책 부합 확인" }
  ```
  - `actor_role`이 `AX_PLAN`이 아니면 → **403**(권한 통제가 실제로 작동). ← 로그인 역할이 여기서 의미를 가짐
  - `decision:"REJECT"`면 **사유 필수**, 없으면 409.
- **백엔드 처리:**
  - 일반 HEAVY(자동화·커넥터, L3 등) → 승인 즉시 **PUBLISHED**
  - **웹앱** → 여기서는 업무·정책 승인만, 실제 빌드는 개발팀 단계에서(아래 4번)
  - 결정은 `AuditLog`에 기록(REVIEW_APPROVE / REVIEW_REJECT + 사유)

**동작 C) 감사 로그 열람**
- **호출:** `GET /audit` → 심사 결정·등급 변경·자동 반려·재심사 진입 등 전 이벤트.

## 4. 관리 — 개발팀 (AX_DEV) 🟢 **[실제 백엔드] — 웹앱 빌드/배포 시뮬**

> 로그인 계정이 AX개발팀(서지훈)일 때. 웹앱 자산은 기획 승인 후 여기서 기술 검증·배포됩니다.

**동작:** 배포 큐에서 "배포 파이프라인 실행".

- **호출:** `POST /assets/{id}/review` `{ "actor_role":"AX_PLAN", "decision":"APPROVE", "reason":"기술 검증·배포" }`
  → 백엔드가 **백그라운드 파이프라인**을 시작(각 단계 `asyncio.sleep`, 기본 1.2초 간격).
- **백엔드 상태 전이(자동):** `REVIEW_PENDING → TECH_REVIEW → BUILD_SIM → DEPLOY_SIM → PUBLISHED`
  - 단계마다 `StatusHistory` + `AuditLog(PIPELINE_*)` 기록
- **프론트가 하는 일:** `GET /assets/{id}`를 폴링하며 **실제 백엔드 상태에 맞춰 8단계 진행 바를 순차 점등**. 마지막 PUBLISHED에서 자산이 실행 URL과 함께 공개.
- **화면 증거:** 파이프라인 로그에 `[backend] status = TECH_REVIEW → BUILD_SIM → DEPLOY_SIM → PUBLISHED`가 시간차로 찍힘 → *"순서대로 점등되는 게 서버 실제 진행"*.

> **왜 시뮬레이션인가:** 과제 범위상 실제 Docker 빌드/배포는 제외(외부 의존성 0·오프라인 시연). 대신 **상태 전이와 감사 로그가 진짜 DB에 남는 것**으로 "비하인드 프로세스 설계"를 증명합니다.

## 부록 — 직접 확인용(서버 켠 상태)

```bash
# 백엔드 실행
cd backend && .\.venv\Scripts\python.exe -m uvicorn main:app --port 8000

# 상태머신 전이가 DB에 남는지 눈으로 확인
curl -s -X POST http://127.0.0.1:8000/demo/reset
curl -s -X POST http://127.0.0.1:8000/assets -H "Content-Type: application/json" \
  -d '{"title":"요약 Gem","type":"AGENT","platform":"GEMINI","data_level":"L2","author":"박서연","dept":"인사팀","purpose":"회의록을 3줄로 요약하는 설정형 에이전트입니다"}'
curl -s http://127.0.0.1:8000/assets/1     # status_history 전체가 보임 = 시연의 근거
```

- 실행/스크립트/상태 다이어그램 상세: `backend/README.md`
- 인증·SSO 확장 설계: `backend/README.md` 8장
